Synopsis of Topics

The module content (see module descriptor) lists topics that we have covered in class. Additionally, there are a few other topics that are also important.

  • Roadblocks in hardware development which led to the development of Instruction Level Parallelism. This involves power consumption, memory bandwidth, efficiency of speed. The limitation of ILP and single-core architectures gave way to multi-core processors. You need to be aware of the hardware components such as different types of memory, their use, limitations, processors, etc.
  • Flynn's Taxonomy - very important classification for hardware-based architectures. Modern architectures, however, are a mix of many of these. It is important to understand the advantages and disadvantages of each category in Flyyn's taxonomy, and why they are needed (for specific tasks).
  • Architectures: scalar/superscalar, vector, VLIW
  • MultiThreading: SMT / Hyperthreading ; Threads & processing cores ; Threading in GPU
  • Instruction Set Architectures: VLIW, Multimedia, SSE/MMX
  • Optimisation: scheduling, out-of-order executions
  • Asynchronous Programming Model
  • Parallel programming with OpenMP

Answering Questions

  • The structure and distribution of questions for the examination is much similar to that of the previous year (and those before).
  • There will be 3 questions of 50 marks each, of which 2 need to be attempted.
  • The previous email has already listed the important topics for the module
  • If there are any questions that require writing code in the exam-
    • writing syntactically correct code would be good, but is not a necessity i.e. missing a few semicolons or brackets is OK as long as the general code is well laid out.
    • do not write obfuscated code
    • do not introduce extra features (and bugs!) that are not part of the question
    • do not use external libraries ; standard (POSIX) libraries are okay to use
    • remember that you have been programming for at least three years now - use understandable variable names, programming styles, etc.
  • For questions that require describing your code or approaches towards the code-
    • do not state the obvious e.g. you used #pragma omp for because there was a for loop; instead, you used it to parallelise a set of repeating statements (which happened to be in a for loop)
    • state the (sometimes hidden) assumptions: did something depend on something else? was your approach safe because the data is never written? State this.
    • describe why you chose a particular construct e.g. why did you chose to use _mm_load_ps instead of _mm_loadu_ps or for openMP section vs task
    • describe why your approach is good / performance / safe
  • I'm not encouraging writing essay-style answers. If possible, be concise and to the point.
  • remember: even if you provide the correct answer - if you don't have evidence / facts / justifcations to back it up, you will probably lose some marks.
