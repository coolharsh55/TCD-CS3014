CS3014 Assignment

The assignment is simple .
Implement 1 searching algorithm and 1 sorting algorithm in non-parallel and parallel execution formats, and compare them.
Write a report on your findings.

Submission date: FRIDAY 23:59 16 March 2018 UTC .

How to submit: use blackboard, upload as many times as you want

Selecting Algorithms

You must explain the reason why you selected a particular algorithm.
You can either choose an algorithm to demonstrate that:

  • it is NOT parallelisable in an efficient way
  • or, you can improve its performance by parallelising it.

Both approaches must be proved using code and performance metrics.

Algorithms must be run on size >= 100000 items.

You may select any sorting algorithm and any searching algorithm of the following:

searching algorithms

  • linear search
  • k^th selection
  • binary search
  • fibonacci search
  • hash table (search)

sorting algorithms

  • cocktail sort
  • insertion sort
  • shell sort
  • quicksort
  • mergesort
  • heapsort

Report

  • MAXIMUM 5 pages
  • Must have two parts - searching & sorting
  • Each part must have the following explanations:
    • why was the algorithm chosen?
    • what is its time complexity?
    • what strategies did you apply when parallelising it?
    • compare performance of serial vs parallel implementations
    • assumption on number of CPU cores and threads
  • There must be no code in the report. You may use snippets for explanation, but snippets may not be longer than 3 lines. Essentially, your report must not be full of code.
  • You must describe the execution environment you chose in the code (how many processes? how many threads?)
  • You must describe the physical environment of your test machine (What CPUs? How many cores? Hyperthreading? How much cache? How much RAM?)
  • You must mention the version of compiler, C/C++ library, OS, etc.
  • You must test your code with different parameters (thread count, for loop count, etc.) - you must explain what effect this has on the code.

Code

  • NOTE: These instructions have been changed
  • You must submit all related code (source files only, no binaries) as part of your submission as an archive file (zip or tar/tar-gz)
  • For additional references to code not part of the assignment, you may use any form of code hosting as long as it is accessible without restrictions. Github, Gitlab, Bitbucket, Pastebucket is OKAY to use.
  • Each approach should have its own file - therefore, at minimum, you will have four files {serial, parallel} x {searching, sorting}
  • You can create a Makefile for executing your code with the following structure:
    • clean: deletes all runtimes and programs
    • searchserial: compiles the serial searching code
    • searchparallel: compiles the parallel searching code
    • sortserial: compiles the serial sorting code
    • sortparallel: compiles the parallel sorting code
    • runserialsearch: run the serial search program
    • runparallelsearch: run the parallel search program
    • runserialsort: run the serial sort program
    • runparallelsort: run the parallel sort program
    • This is optional (but highly recommended)
  • Your code must be documented. Do not put obvious comments. Do not repeat things that are in the report. Put comments that explain the code or the approach as necessary.
  • Your code must be readable. Non-readable code will not be acceptable.
  • Your code must be compilable. Non-compilable code will not be acceptable. Your code must have comments describing how it should be compiled.
  • Code that does not compile will not be graded, not even partially.
  • Use of optimsation compiler flags is NOT allowed.
  • Your program should NOT print any messages to any interface. If these are present for debugging purposes, it is expected that the debug option will be turned off before submission.

Evaluation

Searching - 5 marks (1 marks each)

  • What algorithm did you choose? Why did you choose it?
  • How did you parallelise it? Or how did you prove it was not parallelisable?
  • Parallel code - data access, data constraints
  • Parallel code - effectiveness of parallel constructs
  • Testing and performance metrics

Sorting - 15 marks

  • What algorithm did you choose? Why did you choose it? (1 mark)
  • How did you parallelise it? Or how did you prove it was not parallelisable? (3 marks)
  • Did you break the algorithm down into blocks that were parallelisable and not parallelisable? (2 marks)
  • Did you consider where parallel constructs might need to be combined/joined/waited? (2 marks)
  • Did you effectively share variables amongst threads/processes? (2 marks)
  • Did you take care to avoid the problems of accessing same data in multiple parallel constructs? (2 marks)
  • Testing and performance metrics (3 marks)

Areas of concern where marks may be deducted:

  • Your report does not contain correct explanations.
  • You did not repeat tests with multiple parameters.
  • You did not optimise for data access in loops, etc.
  • You did not document the approach
  • You did not take your environment (OS, CPU cores, cache) into account

do's

  • use OpenMP, and optionally you can also use MPI / Intel_SSE
  • use C - no other programming languages; C++ is okay if using only standard libraries
  • use gdb / valgrind / purify (for debugging)
  • look up implementations online (with citations)
  • discuss problems, discuss approaches

dont's

  • don’t cheat - suspicious similarity in code, metrics, evaluations, results, and reports will be a cause for YOUR concern (for marks)
  • use any external library except those provided by default in a POSIX environment such as Linux or UNIX (sub)systems
  • select exotic algorithms
  • copy code verbatim (you must learn and adapt it on your own)
  • use code / approaches / explanations without citations
  • copy code / approaches / reports from each other
  • post the assignment on stackoverflow or share it otherwise
